## 策略：渐进式重写，而非大爆炸

一次性重写风险太高，建议**核心用 Rust，业务层保留 Python**。

---

## 第一阶段：架构拆分

```
┌─────────────────────────────────────────┐
│              OpenClaw Core (Rust)       │
├─────────────────────────────────────────┤
│  Gateway  │  Message Router  │  Cron   │
│  (ws服务)  │  (多平台消息)      │  (调度) │
└─────────────────────────────────────────┘
                      │
              IPC / gRPC
                      │
┌─────────────────────────────────────────┐
│         Tool Layer (Python)             │
│  browser │ exec │ file │ llm │ mcp...  │
└─────────────────────────────────────────┘
```

---

## 第二阶段：优先级排序

| 优先级 | 模块 | 理由 |
|--------|------|------|
| P0 | Gateway (ws/消息路由) | 性能瓶颈在 I/O |
| P1 | Browser CDP Client | 稳定连接管理 |
| P2 | Cron/Heartbeat | 定时精度要求高 |
| P3 | Tool 原生实现 | 逐步迁移 |

---

## 第三阶段：关键决策

1. **Async Runtime**: tokio（生态最成熟）
2. **FFI 方案**: PyO3 绑定现有 Python 工具，或直接用 Rust 实现
3. **配置管理**: toml + serde，放弃 JSON
4. **插件系统**: 用 Rust trait 动态加载
