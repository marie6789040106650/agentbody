## 🤖 子代理创建规则

### 核心理念

**自主判断**: 根据任务复杂程度，自主决定是否创建子代理

### 判断标准

| 任务特征 | 创建子代理 | 说明 |
|----------|------------|------|
| 耗时 > 5 分钟 | ✅ | 后台执行，不阻塞主流程 |
| 需要持续监控 | ✅ | 如 LLVM 下载 |
| 独立可执行 | ✅ | 无需人工干预 |
| 并行任务 | ✅ | 多任务同时进行 |
| 简单任务 (< 5分钟) | ❌ | 直接执行 |
| 需要交互 | ❌ | 主代理处理 |
| 决策类 | ❌ | 主代理处理 |

### 自主创建流程

```
任务到来
    ↓
判断复杂度
    ↓
┌─────────────────────────────────────┐
│ 复杂任务 (>5分钟/需监控/并行)?       │
│ ├─ 是 → 创建子代理                  │
│ └─ 否 → 直接执行                   │
└─────────────────────────────────────┘
    ↓
    ↓ 创建子代理
    ├── 规划任务目标
    ├── 定义验收标准
    └── 记录到 SUB_AGENTS_PLAN.md
```

### 示例

| 任务 | 复杂度 | 决策 | 原因 |
|------|--------|------|------|
| LLVM 下载监控 | 高 | ✅ 子代理 | 需持续监控数小时 |
| .so 构建 | 高 | ✅ 子代理 | 耗时 30 分钟 |
| 部署程序 | 中 | ✅ 子代理 | 可独立执行 |
| 整理文档 | 低 | ❌ 直接执行 | 只需 5 分钟 |
| 规划任务 | 低 | ❌ 直接执行 | 决策类 |

### 验收标准模板

```markdown
**验收标准**:
- [ ] 产出文件/结果
- [ ] 验证命令
- [ ] 主代理确认
```

## 📊 子代理模型动态选择

### 两种模型对比

| 特征 | isolated session | system event |
|------|------------------|--------------|
| 上下文 | 独立隔离 | 共享主会话 |
| 超时限制 | 无 (可配置) | 受主会话限制 |
| 结果获取 | 需手动获取 | 立即返回 |
| 内存使用 | 独立 | 共享 |
| 并行能力 | 强 | 受限 |

### 动态选择标准

| 任务特征 | 推荐模型 | 原因 |
|----------|----------|------|
| 耗时 > 5分钟 | isolated | 避免超时 |
| 需要持续监控 | isolated | 后台运行 |
| 独立任务 | isolated | 隔离执行 |
| 耗时 < 2分钟 | system event | 立即结果 |
| 依赖主会话 | system event | 共享上下文 |
| 需要并行 | isolated | 独立运行 |

### 实际应用示例

| 子代理 | 任务 | 模型 | 原因 |
|--------|------|------|------|
| monitoring-agent | LLVM 监控 | isolated | 持续运行 |
| build-agent | .so 构建 | isolated | 耗时 30分钟 |
| deploy-agent | 部署 | system event | 快速 <1分钟 |
| test-agent | 测试 | isolated | 耗时 10分钟 |
| domain-agent | 域名注册 | isolated | 需持续交互 |

## 🔬 每周研究机制

### 研究频率

| 周期 | 时间 | 任务 |
|------|------|------|
| 每周一 | 09:00 | 研究新功能/最佳实践 |
| 每周四 | 14:00 | 复盘+迭代优化 |

### 研究主题轮换

| 周 | 主题 | 重点 |
|----|------|------|
| 1 | 效率提升 | 工具优化、缓存策略 |
| 2 | 自动化 | Cron、工作流、节点 |
| 3 | 协作模式 | 多代理、子代理优化 |
| 4 | 高级功能 | Gateway、插件、安全 |

### 研究方式

**双模型并行研究**:
- 子代理 A: MiniMax (复杂研究任务)
- 子代理 B: Qwen (快速调研任务)
- 主代理: 结果汇总

### 实施计划

| Phase | 时间 | 内容 |
|-------|------|------|
| Phase 1 | 本周 | 启用 Gateway 缓存、配置子代理超时、优化心跳机制 |
| Phase 2 | 下周 | 配置 Cron 调度、配置工作流自动化、部署节点自动化 |
| Phase 3 | 下月 | 多代理协作、任务路由、性能监控 |

## 📋 研究协议与机制

### 研究策略

**核心原则**: 先全面，后迭代

| 轮次 | 目标 | 说明 |
|------|------|------|
| **第一轮** | 全面了解 | 4 个主题全部深入研究一遍 |
| **第二轮+** | 迭代优化 | 基于第一轮经验进行深化 |

### 研究顺序

| 顺序 | 主题 | 目标 |
|------|------|------|
| 1 | 效率提升 | 建立基准 |
| 2 | 自动化 | 建立流程 |
| 3 | 协作模式 | 建立机制 |
| 4 | 高级功能 | 建立能力 |
| 5+ | 迭代优化 | 持续提升 |

### 定时启动机制

| 触发方式 | 优先级 | 说明 |
|----------|--------|------|
| **Cron 触发** | 高 | 定时启动 |
| **心跳触发** | 中 | 心跳时检查并启动 |
| **手动触发** | 低 | 随时可以启动 |

**遗漏补偿**:
- 当天内: 心跳时检测并启动
- 超过1天: 立即启动，不等待
- 超过3天: 简化研究，快速完成

**核心原则**: 宁可启动，不要等待
